## 2026-06-20 · Claude (Opus 4.8) · 修复 + 全面审计

本次由次开发 Claude 接手排查「登录 403」，并扩展为一轮完整审计。后端在本地真实跑起来（便携 JDK+Maven+MariaDB），所有结论均经**实际打接口验证**，非纯静态推断。

### 一、登录 403 排查与修复（主诉求）

朋友反馈「登录就 403，查了跨域/JWT 过滤器/安全配置/密码校验都没问题」。实际根因不在他查的那几层。共发现 **3 个串联的 Bug**：

**Bug 1（403 根因）· `SecurityConfig` 路径前缀错误**
- 现象：`POST /api/auth/login` 返回裸 403（无 JSON body）。
- 根因：`application.yml` 配了 `server.servlet.context-path: /api`，而 `SecurityConfig` 的 `antMatchers` 写成了 `/api/auth/**`。Spring Security 匹配的是**剥掉 context-path 之后**的路径（`/auth/login`），`/api/auth/**` 永远匹配不上 → 登录落到 `anyRequest().authenticated()` → 未认证 → 进 Controller 前由 `Http403ForbiddenEntryPoint` 抛 403（因此**不经过** `GlobalExceptionHandler`，body 为空）。
- 旁证：同段配置里 swagger 的 matcher 写的是不带 `/api` 的（正确），auth/notices 却带了 `/api` —— 作者当时对 context-path 的理解不一致。
- 修复：`/api/auth/**` → `/auth/login`；`/api/notices/public/**` → `/notices/public/**`。
- 注：朋友给 `JwtAuthenticationFilter` 加的「跳过 `/api/auth/`」用的是 `getRequestURI()`（**含** `/api`），判断本身没错，但过滤器从不拦截登录（登录无 token，过滤器只是不设认证就放行），所以改过滤器治不了病。本次把它收敛为只跳过 `/api/auth/login`，使 `/auth/me` 仍能正常校验 token。
- 文件：`security/SecurityConfig.java`、`security/JwtAuthenticationFilter.java`

**Bug 2 · 初始管理员密码哈希是错的**
- 根因：`sql/schema.sql` 中 `admin` 的 BCrypt 哈希实测与 `admin123` **不匹配**（`checkpw=false`，疑似手工凑长度的假哈希）。即便 403 修好，也会卡在「密码错误」。
- 修复：重新生成正确的 `$2a$10$` 哈希写入 schema.sql。**已导入的库需执行**：
  ```sql
  UPDATE sys_user SET password='$2a$10$Bu1o08vpQfeg9j7dv7YL/.mju334e1qbXuP8sUZBOpfdhiB1aOVU6' WHERE username='admin';
  ```
- 文件：`sql/schema.sql`

**Bug 3 · 密码错误返回 500 而非 401**
- 根因：`GlobalExceptionHandler` 没有处理 Spring Security 的 `BadCredentialsException`，掉进兜底 `Exception` → 500「系统繁忙」。
- 修复：新增 `@ExceptionHandler(AuthenticationException.class)` → 401「用户名或密码错误」；并改进 `admin-web/src/utils/request.js` 错误拦截器，优先展示后端 `msg`（原先任何错误只显示 "Request failed with status code XXX"）。
- 文件：`common/GlobalExceptionHandler.java`、`admin-web/src/utils/request.js`

**验证**：`admin/admin123` → 200 + JWT；错误密码 → 401 + 「用户名或密码错误」；无 token 访问受保护接口 → 403。✅

用 admin token 把全部 Controller 的 GET 接口扫了一遍，再针对失败项查源码 + 看堆栈。

**已修复：**

**Bug 4 · `/admin/dashboard` 一调就崩**
- 根因：`dashboard(RepairOrderRepository repairRepo)` 把 Repository 错误地声明为**请求绑定的方法参数**（Spring MVC 试图从请求里绑定它），调用其方法时 Spring Data 抛 "is no accessor method"；外加一行占位垃圾 `VisitorRepository.class.getClass()`。此接口从未成功过。
- 修复：改为构造注入 `RepairOrderRepository`/`VisitorRepository`；给 `VisitorRepository` 加 `countByStatus`；返回真实统计 `{totalOwners, totalEmployees, pendingRepairs, pendingVisitors}`。实测 200。
- 文件：`controller/AdminController.java`、`repository/VisitorRepository.java`

**Bug 5 · 客户端错误一律 500**
- 根因：缺必填参数 / 参数类型不符 / 请求体 JSON 格式错，均掉进兜底 `Exception` → 500。
- 修复：`GlobalExceptionHandler` 新增对 `MissingServletRequestParameterException` / `MethodArgumentTypeMismatchException` / `HttpMessageNotReadableException` 的处理 → 400。实测缺参数从 500 变 400。
- 文件：`common/GlobalExceptionHandler.java`

## 评审建议（给 DS）

> 次开发视角的建议，按优先级。DS 采纳后请到《主开发文档》第 8 节勾掉对应项。

1. 🔴 **优先修越权(IDOR)**：`my-orders / my-visitors / butler messages` 等「我的 xx」接口不要信任前端传的 `ownerId/roomId`，应从 `@CurrentUser SysUser` 取当前登录人的 id。这是目前最实质的安全风险。
2. 🟠 **清理分页双层包裹**：删 `PageResult` 的 `code/msg`，让响应回到单层 `Result{code,msg,data:{total,page,size,list}}`。admin-web 读 `.list` 不受影响，**改前确认 miniapp 的读法**。
3. 🟠 **后台 Dashboard 接真实接口**：`Dashboard.vue` 现为写死假数据，后端 `/admin/dashboard` 已可用，接上即可。
4. 🟠 **安全加固**：默认密码 `123456` → 随机或强制首登改密；`application.yml` 的 JWT 密钥 / DB 密码移到环境变量，勿入库。
5. 🟢 **统一约定**：分页基准统一为 1-based 并在前端封装；admin 列表接口补服务端分页。
6. 🟢 **错误码体系**：`Result` 用 `code` 同时 `@ResponseStatus` 设 HTTP 状态，二者语义已基本对齐，后续新增接口沿用即可。
7. 💡 **建议习惯**：因 `ddl-auto:none`，改实体字段务必同步 `schema.sql`；改 `SecurityConfig` 前重读《主开发文档》第 5 节的 context-path 约定，避免再次踩 403。

## 2026-06-20 · Reasonix (DeepSeek v4 Pro) · 修复（处理 Claude 评审建议）

次开发 Claude 留下的 7 条评审建议全部处理完毕。

### 一、🔴 IDOR 越权修复（建议 1）

**修复范围**：5 个后端 Controller + 10 个前端文件

| 端点 | 原问题 | 修复方式 |
|---|---|---|
| `GET /repair/my-orders` | 接受前端传 `ownerId`，可越权查看他人报修 | 改为 `@CurrentUser SysUser`，强制取 `user.getOwnerId()` |
| `GET /visitor/my-visitors` | 同上 | 同上 |
| `GET /butler/messages` | 接受前端传 `roomId`，业主可看他人消息 | OWNER 角色强制 `roomId = user.getOwnerId()`；EMPLOYEE/ADMIN 可传参 |
| `GET /fee/bills` | 接受前端传 `ownerId` | OWNER 角色强制取自身账单；EMPLOYEE/ADMIN 可传参 |
| `GET /convenience/usage-records` | 同上 | 同上 |

**前端同步**：miniapp 的 `api/repair.js`、`api/visitor.js`、`api/butler.js`、`api/fee.js`、`api/convenience.js` 均移除 `ownerId`/`roomId` 参数（后端从 token 推导）。对应 5 个页面调用处同步更新。

### 二、🟠 去掉分页双层包裹（建议 2）

- 删除 `PageResult.java` 的 `code`/`msg` 字段，响应回归单层 `Result { code, msg, data: { total, page, size, list } }`。
- 确认 admin-web 和 miniapp 均只读 `.list`/`.total`/`.page`/`.size`，不受影响。

### 三、🟠 后台 Dashboard 接真实接口（建议 3）

- `admin-web/src/api/admin.js` 新增 `getDashboard()`。
- `Dashboard.vue` 改为 `onMounted` 时调用 `/admin/dashboard`，显示实时 `totalOwners`、`totalEmployees`、`pendingRepairs`、`pendingVisitors`。

### 四、🟠 安全加固（建议 4）

- `AdminController.createOwner/createEmployee` 添加 TODO 标注默认密码 `123456` 应随机化。
- `application.yml` 的 JWT 密钥和 DB 密码添加 TODO 标注应迁移到环境变量。
- `JwtAuthenticationFilter` 添加 TODO 标注 `loadUserById` 应考虑 Redis 缓存。

### 五、🟢 分页基准统一 + admin 列表补分页（建议 5）

- `AdminController.getOwners`/`getBuildings`/`getEmployees` 改为接受 `page`/`size` 参数，返回 `PageResult`，服务端 1-based 分页。
- 注入 `OwnerRepository`/`BuildingRepository` 直接使用 Spring Data 分页。
- admin-web 三个视图（Owners/Buildings/Employees）同步添加 `el-pagination` 组件。
